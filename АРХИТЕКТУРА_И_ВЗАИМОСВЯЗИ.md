# Архитектура и взаимосвязи проекта

## 1. Общая логика системы

Текущий pipeline:

`Микрофон -> ASR -> NLU -> Executor -> Лог/GUI/TTS`

Если NLU не нашел локальный интент, подключается fallback-ветка:

`Local fallback -> AI fallback`

Ключевая идея проекта:  
**сначала локальная надежная логика, AI — как дополнительный уровень гибкости.**

## 2. Основные модули

### `src/jarvis/engine.py`

Главный оркестратор голосового цикла:

- запуск/остановка движка;
- wake-word (Porcupine) и continuous-режим;
- вызов ASR и получение текста;
- маршрутизация в NLU/Executor;
- блокировка микрофона во время TTS;
- запуск continuous-таймера после завершения озвучки;
- логирование технических событий.

### `src/jarvis/nlu.py`

Rule-based распознавание интентов:

- шаблоны и regex для локальных команд;
- извлечение параметров (`slots`);
- нормализация некоторых STT-ошибок;
- отдельные правила для сценариев (`run_scenario`).

Возвращает:

`{"type": "<intent>", "slots": {...}}`

### `src/jarvis/executor.py`

Центр исполнения:

- `run(intent)` — единая точка действий;
- работа с `apps`, `sites`, `synonyms`, `scenarios`;
- домены: задачи, таймеры, напоминания, погода, окна, медиа, системные действия;
- запуск пользовательских сценариев (`open:`, `url:`, `bat:`);
- обработка неизвестных команд (`handle_unrecognized_command`);
- подтверждение рискованных действий;
- запуск `.exe` с корректным `cwd` (исправление для OBS).

### `src/jarvis/unified_ai_turn.py`

Контракт AI-ответа:

- режимы `reply` и `action`;
- парсинг/валидация JSON-структуры;
- ограничение небезопасных действий.

### `src/jarvis/memory_store.py`

Память о пользователе:

- хранение списка фактов;
- выборка и удаление;
- контекст для AI-диалога.

### `gui/app.py`

Интерфейс и настройка:

- главный экран (аватар, лог, старт/стоп);
- экран настроек;
- разделы микрофона, распознавания, TTS, AI, конфигурации;
- менеджеры `Программы`, `Сайты`, `Сценарии`;
- конструктор сценариев (шаги + редактирование).

## 3. Детальный поток команды

1. Пользователь говорит команду.
2. ASR распознает текст.
3. `engine` вызывает `nlu.parse(...)`.
4. Если интент локальный -> `executor.run(intent)`.
5. Если `unknown`:
   - сервисные локальные проверки (память, контекст, уточнения),
   - local-domain fallback,
   - unified AI fallback.
6. Исполнение действия.
7. Лог + озвучка результата.

## 4. Открытие программ и сайтов

Для команд "открой/запусти":

1. резолв через `synonyms`;
2. проверка `apps` и `sites`;
3. если не найдено — fallback в AI/web ветку.

Дополнительно реализовано:

- мягкий матч для коротких STT-ошибок (`об са` -> `обс` -> `obs`);
- корректный `cwd` при запуске `.exe`.

## 5. Сценарии

Хранилище: `config.json -> scenarios`.

Формат шагов:

- `open:<app_key>`
- `url:<https://...>`
- `bat:<path>`

GUI-поддержка:

- создание;
- редактирование;
- удаление;
- выбор `.bat/.cmd` с копированием в `data/scenario_scripts`.

## 7. Безопасность и ограничения

- Опасные команды (`shutdown/restart/sleep`) требуют подтверждения:
  - `подтверждаю`
  - `отмена`
- AI не должен напрямую выполнять системные рискованные операции без проверки.
- Приоритет всегда у локальной логики.

## 8. Почему архитектура удачна для диплома

- модульная и простая;
- легко демонстрируется на живом примере;
- понятно, где rule-based слой, где AI-слой;
- легко расширять: новый интент = правило в NLU + обработчик в Executor + тест.
