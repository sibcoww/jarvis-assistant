# Архитектура проекта и взаимосвязи модулей

## 1. Общая схема

Архитектура построена как последовательный конвейер:

`Микрофон -> ASR -> NLU/AI -> Executor -> Системное действие -> Лог/TTS/GUI`

Где:

- **ASR** — преобразует речь в текст.
- **NLU** — локально определяет тип команды (интент) и параметры (slots).
- **AI fallback** — подключается, если локальные правила не покрыли фразу.
- **Executor** — единая точка выполнения всех действий.

## 2. Ключевые модули

## `src/jarvis/engine.py`

Главный оркестратор:

- управляет запуском/остановкой движка;
- обрабатывает wake-word/continuous/PTT;
- получает текст от ASR;
- вызывает NLU и `Executor`;
- контролирует подтверждение опасных действий;
- запускает TTS и фоновые проверки таймеров/напоминаний;
- пишет служебные логи.

## `src/jarvis/nlu.py`

Локальный rule-based NLU:

- распознает интенты по ключевым словам и регулярным выражениям;
- извлекает числа (`extract_number`);
- нормализует повторяющиеся слова STT (`collapse_repeated_stt_words`);
- возвращает структуру:

`{"type": "<intent>", "slots": {...}}`

## `src/jarvis/executor.py`

Центр бизнес-логики:

- валидирует AI-команды;
- хранит контекст короткой истории диалога;
- реализует локальные домены (todo/timer/reminder/weather/system/windows/presentation);
- выполняет действия через `run(intent)`;
- ведет историю действий и поддерживает повтор последней команды;
- управляет подтверждением рискованных операций.

## `src/jarvis/unified_ai_turn.py`

Контракт взаимодействия с моделью:

- задает системный prompt;
- стандартизирует формат AI-ответа;
- разделяет режимы `reply` и `action`.

## `src/jarvis/memory_store.py`

Долгосрочная память пользователя:

- хранит список фактов;
- добавляет/удаляет/ищет факты;
- формирует контекст для AI.

## `gui/app.py`

Графический интерфейс:

- кнопки управления движком;
- отображение логов;
- настройки;
- tray-уведомления;
- интеграция с TTS логикой.

## 3. Поток выполнения команды

1. Пользователь говорит фразу.
2. `engine.py` получает текст из ASR.
3. `nlu.py` пытается распознать интент локально.
4. Если `unknown`, `executor.handle_unrecognized_command(...)`:
   - сначала локальные служебные ветки (память, очистка контекста, уточнения);
   - затем local AI fallback для локальных доменов;
   - затем unified AI (общая ветка), если допустимо.
5. Полученный интент идет в `executor.run(...)`.
6. `run(...)` вызывает конкретный метод действия.
7. Результат отражается в логах, GUI и TTS.

## 4. AI fallback стратегия

Проект использует гибрид:

- **Локальные правила** — быстрые и стабильные;
- **AI fallback** — для естественных фраз, когда шаблон не совпал.

Для локальных доменов используется отдельная строгая валидация AI-JSON, чтобы снизить ложные веб-действия.

## 5. Безопасность действий

Опасные системные команды (`shutdown/restart/sleep`) требуют подтверждения:

- `подтверждаю` — выполнить;
- `отмена` — отменить.

Это реализовано через `pending_confirmation` в `executor.py` и обработку шага подтверждения в `engine.py`.

## 6. Хранилища данных

Локальные JSON-файлы в `~/.jarvis/`:

- `chat_history.json` — короткая история диалога;
- `user_memory.json` — факты о пользователе;
- `notes.json` — заметки;
- `reminders.json` — напоминания;
- `todos.json` — задачи.

## 7. Почему архитектура подходит для диплома

- простая и модульная;
- легко расширяется новыми интентами;
- разделены роли: оркестрация, понимание, выполнение, UI;
- понятный баланс: локальная надежность + AI-гибкость.

## 8. Как расширять проект дальше

Добавление новой команды делается в 4 шага:

1. Добавить правило в `nlu.py` (интент + slots).
2. Добавить маршрут и метод в `executor.py`.
3. При необходимости — добавить local AI fallback валидацию.
4. Добавить тесты в `tests/test_nlu.py` и `tests/test_executor.py` (и при AI — `tests/test_executor_ai.py`).
